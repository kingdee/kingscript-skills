# 代码审查模板

适用于“审查一段 Kingscript 脚本是否存在 bug、风险、行为回归或边界遗漏”的场景。

## 推荐输出结构

### 1. 审查范围

- 脚本名称、插件类型、关键事件
- 影响对象：表单、单据、列表、报表、操作服务或调度

### 2. 主要发现

- 优先列 bug、风险、行为回归
- 每条发现都要尽量说明触发条件和影响

### 3. 兼容性关注点

- 是否依赖未开放的 SDK
- 是否忽略权限、租户、账套、组织、上下文边界
- 是否存在生命周期选错、插件基类选错、参数类型不匹配

### 4. 建议动作

- 哪些必须修
- 哪些建议补充验证
- 哪些地方需要补文档或补示例

## 审查时优先检查

- 是否存在行为回归风险
- 是否依赖未开放的后端能力
- 是否遗漏异常处理
- 是否忽略租户、账套、权限边界
- 是否需要补充回归验证点

## 已填充示例

下面给出一个高频审查场景：某个插件在 `registerListener` 里注册监听器时，按钮点击注册值与 `itemClick` 中判断值不一致，导致页面点击后不触发预期逻辑。

### 审查范围

- 插件类型：单据插件
- 核心事件：
  - `registerListener`
  - `itemClick`
- 影响对象：采购订单页面上的自定义工具栏按钮

### 主要发现

- 按钮监听注册值与点击判断值不一致时，页面会出现“按钮能显示，但点击无效”的行为问题
- 如果开发者在 `registerListener` 里遗漏 `super.registerListener(e)`，父类监听逻辑可能失效
- 如果一次性注册过多监听但没有按场景做 null 判断，后续切换表单布局或控件可见性时容易出错

### 示例问题代码

```typescript
registerListener(e: any): void {
  super.registerListener(e);
  this.addItemClickService("btn_batch_import");
}

itemClick(e: any): void {
  super.itemClick(e);

  if (e.getItemKey() === "btn_price_query") {
    this.getView().showTipNotification("打开价格查询弹窗");
  }
}
```

### 审查结论

1. `registerListener` 注册的是 `btn_batch_import`，但 `itemClick` 判断的是 `btn_price_query`，两者不一致，按钮事件不会按预期命中。
2. 当前代码没有说明这两个按钮是否都真实存在，存在配置与脚本不一致的风险。
3. 如果页面只配置了其中一个按钮，另一条分支会形成无效死代码。

### 兼容性关注点

- 工具栏控件标识必须和页面元数据中的按钮标识一致
- 如果按钮是动态加的，还要确认是否在正确生命周期中完成注册
- 如果点击后还要打开子页面或查后端服务，需继续检查权限和上下文前提

### 建议动作

- 必须修：统一 `registerListener` 与 `itemClick` 中的按钮标识
- 必须修：确认页面元数据里真实存在这些按钮
- 建议补充：对打开页面、查价格、查供应商等后续分支增加最小回归验证
