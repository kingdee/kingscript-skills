# Kingscript Concepts

本文件用于沉淀稳定的领域背景，不放具体实现细节。

## Kingscript 是什么

Kingscript 是面向苍穹平台二开场景的脚本化扩展能力。它让二开人员在 VSCode、WebIDE 等工具中使用接近 TypeScript 的开发体验，为表单、单据、列表、报表、操作服务等扩展点编写脚本插件，并在运行时通过 GraalJS 调用后端已经开放的 Java 能力。

它不是一个通用前端框架，也不是浏览器端脚本体系，而是服务于苍穹元数据驱动平台的定制化脚本层。

## 为什么会有 Kingscript

在传统 Java 插件模式下，只要某个公共方法对外可见，二开代码就可能直接依赖它。这会给平台演进带来较强的二开兼容性负担，很多原本只想内部复用的方法一旦被外部使用，后续就很难调整。

Kingscript 的核心思路是：

- 平台研发通过注解显式声明“哪些类和方法可以开放给二开”
- 构建阶段扫描这些开放能力，生成二开可见的 TypeScript 声明
- 二开人员只围绕这些已开放能力编写脚本
- 运行时再把脚本调用路由到后端真实 Java 实现

这样做的目的不是单纯“换一种写代码方式”，而是把开放边界收紧、把兼容性责任前移到 SDK 生成环节。

## 与苍穹平台和元数据模型的关系

苍穹平台本身是元数据驱动的低代码平台，用户可以通过设计器拖拉拽配置单据、表单、字段和页面布局。Kingscript 运行在这个平台模型之上，主要负责补足设计器难以表达的特殊业务逻辑。

可以把三层关系理解为：

1. 元数据层：定义单据、字段、布局、按钮、页面结构
2. 脚本层：在既定扩展点上编写 Kingscript，补充校验、联动、服务调用、页面行为
3. Java 开放层：平台通过 SDK 注解显式开放的后端能力，供脚本在运行时调用

Kingscript 不是替代元数据配置，而是增强元数据配置。

## 与 Java 插件模式的区别

### Java 插件

- 适合平台内部或强控制环境
- 可直接访问较完整的后端类能力
- 兼容性压力更大
- 二开依赖边界不够清晰时，平台升级成本高

### Kingscript 插件

- 面向二开和可控开放场景
- 只允许使用已开放的 SDK 能力
- 更适合做边界清晰、受约束的定制化开发
- 编辑期依赖 TypeScript 声明，运行时仍受平台上下文和 Java 能力约束

## TypeScript 声明与运行时的关系

Kingscript 的开发体验建立在 TypeScript 声明之上，但运行时并不是浏览器里的 TypeScript，也不是 Node.js。

需要牢记三件事：

1. `.d.ts` 只是“可见能力声明”，不是运行时绝对保证
2. 真正能不能调用，取决于后端是否真的开放了该能力、当前场景是否允许、入参类型是否匹配
3. 运行时很多对象本质上仍然是 Java 对象，而不是纯 JavaScript 对象

这也是为什么在 Kingscript 里经常会遇到：

- 有声明但运行时报 `method not found`
- `QFilter`、`ArrayList`、`BigDecimal`、`Date` 等类型不能完全按原生 JS 习惯来用
- 接口 default 方法、集合类型、日期比较方式与普通 TS/JS 认知不完全一致

## 典型二开场景

Kingscript 最常见的使用场景包括：

- 按钮点击处理
- 字段联动和表单事件
- 保存前后校验
- 调用后端开放服务查数、保存、删除或执行业务逻辑
- 列表、报表、操作服务等插件扩展
- 页面提示、弹窗、子页面回调
- 查询条件构造、数据转换和异常处理

这类场景的共同点是：需要基于平台上下文，在明确的扩展点里补充业务逻辑。

## 常见上下文与生命周期心智模型

做 Kingscript 二开时，至少要同时考虑三类上下文：

- 场景上下文：当前是表单、单据、列表、报表、操作服务，还是其他插件类型
- 生命周期上下文：当前处于初始化、注册监听、数据绑定前后、属性变更、保存前后、回调返回等哪个阶段
- 运行时上下文：当前用户、组织、租户、账套、权限、数据状态是否满足调用前提

很多问题并不是“代码写错”，而是：

- 选错了插件基类
- 事件时机太早或太晚
- 当前上下文没有所需对象
- 运行时权限或隔离边界不满足

## 适用对象

- Kingscript 二开开发者
- 平台研发和 SDK 开放能力维护者
- 实施顾问、解决方案顾问
- 需要借助 AI 快速生成、修改、排障脚本的人

## 不适用场景

Kingscript skill 不应被当作：

- 平台真实行为的唯一权威来源
- 私有 API 猜测器
- 绕过权限、租户、账套、组织边界的工具
- 忽略运行时差异、只按 `.d.ts` 生搬硬套的代码生成器

如果文档、声明、示例和运行时表现冲突，应优先承认不确定性，再回到 SDK、Javadoc、日志和实际环境核实。

## 编写要求

- 先讲事实，再讲推论。
- 区分“平台机制”与“当前项目约定”。
- 不写任何未经确认的私有 API。
