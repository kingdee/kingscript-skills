# Kingscript 排障手册

本文件用于沉淀 Kingscript 二开中的统一排查路径，帮助使用者先定位问题类别，再进入具体 FAQ 或代码修复。

## 适用范围

- VSCode 或在线编辑器开发、调试失败
- SDK 类型声明存在，但运行时报错
- Java 对象和 Kingscript 类型交互异常
- 插件上传成功，但表单、单据、报表等场景无法正常绑定或执行
- 同一段代码在不同环境、不同账套、不同上下文下表现不一致

## 总体排查顺序

1. 先确认现场
   - 是哪个业务对象：表单、单据、列表、报表、调度、流程还是控件扩展
   - 是哪个事件或生命周期：如 `propertyChanged`、按钮点击、保存前后、调试启动
   - 报错发生在编辑期、保存脚本时、运行时还是部署后
   - 是否能提供最小复现代码、报错截图、调用的 SDK 或开放类名称
2. 再确认环境
   - 苍穹版本是否支持当前 Kingscript 能力
   - 当前账套、租户、用户登录态是否正确
   - 本地 `node_modules` 是否完整，VSCode 插件是否正常工作
   - 服务是否完成引擎初始化，调试心跳页是否已正常建立
3. 再确认绑定关系
   - 插件基类是否与注册场景匹配
   - 脚本是否上传到了正确位置
   - 表单、单据、报表、调度等场景是否绑定了正确的插件类型
4. 再确认 SDK 与类型
   - 调用的方法是否真的属于已开放 SDK
   - TypeScript 声明是否和运行时能力一致
   - Java 对象、JS 原生对象、BigInt、Date、QFilter 等类型是否用对
5. 最后做回归验证
   - 只改一个因素后重新验证
   - 同时验证编辑器提示、脚本保存、运行结果、日志输出、调试链路

## 通用排查清单

### 1. 编辑期问题

典型现象：
- `import` 无法识别
- 类型提示缺失
- 保存脚本时报方法不存在或接口不兼容

优先检查：
- 本地 `node_modules` 是否已正确拉取
- 当前苍穹版本是否支持 Kingscript
- `import` 路径、包名、大小写是否正确
- 当前调用的方法是否只是声明层可见，运行时未必可用

### 2. 运行期问题

典型现象：
- 调用时报未定义、类型转换异常、序列化失败
- 同样的代码在某些数据上正常，在另一些数据上报错

优先检查：
- 入参是否为 Java 对象还是 JS 原生对象
- `BigInt`、`Date`、`BigDecimal`、集合类是否按 Kingscript 规则使用
- `QFilter`、查询参数、集合参数是否用了正确的 Java 类型
- 是否存在 null、空集合、空上下文、未登录等前置条件缺失

### 3. 绑定与场景问题

典型现象：
- 脚本能上传，但页面选不到
- 绑定成功，但事件不触发
- 同一插件在表单可用，在单据不可用

优先检查：
- 插件基类是否和场景匹配
- 事件是否属于当前插件类型的生命周期
- 脚本上传位置、注册位置、绑定位置是否一致

### 4. 调试链路问题

典型现象：
- VSCode 调试启动失败
- 心跳页打不开或计数不增长
- 已经打断点，但页面操作进不来

优先检查：
- 调试前是否已登录目标环境
- 心跳页域名是否与登录环境一致
- 打开心跳页所用浏览器是否与当前登录环境一致
- 当前场景是否属于同步调试支持范围

### 5. 平台与服务问题

典型现象：
- 引擎未初始化
- 同一段代码昨天正常，今天整体不可用
- 下载依赖、初始化 SDK、执行脚本同时异常

优先检查：
- 服务是否刚启动且初始化尚未完成
- `monitor` 日志中是否存在 `lib` 包缺失或 Kingscript 关键字异常
- 系统补丁、jar 包、运行环境是否完整

## 高频问题专题

### 1. 依赖包无法识别

常见现象：
- VSCode 中 `import` 报红
- 自动补全缺失
- 新建项目后依赖目录为空

可能根因：
- 苍穹版本过低
- 包名或路径拼写错误
- `node_modules` 未成功拉取
- 本地网络拦截下载请求

建议排查顺序：
1. 检查版本是否在支持范围内
2. 检查 `import` 拼写
3. 检查本地 `node_modules`
4. 如为空，重新连接账套拉取依赖
5. 如仍失败，检查 VSCode 网络请求是否被拦截

修复建议：
- 重新初始化本地工程
- 确认网络可访问依赖下载链路
- 不要在依赖未拉取完整时直接判断 SDK 不存在

回归验证：
- 包路径可正常补全
- `import` 不再报红
- 最小示例可成功保存并运行

### 2. 引擎未初始化

常见现象：
- 执行脚本时报引擎池未初始化
- 平台刚启动后大量脚本暂时不可用

可能根因：
- 服务尚未完成初始化
- `lib` 模块缺失
- 系统补丁、jar 包不完整

建议排查顺序：
1. 等待服务初始化完成后重试
2. 查看 `monitor` 日志中的 Kingscript 相关异常
3. 重点搜索 `lib` 缺失、初始化失败等关键字

修复建议：
- 补齐缺失模块或更新补丁
- 必要时重启服务并重新验证初始化链路

回归验证：
- 相同脚本可正常执行
- 同环境下其他 Kingscript 能力也恢复正常

### 3. 无法启动调试

常见现象：
- VSCode 点击调试后直接失败
- 心跳页未打开、打不开或计数不正常

可能根因：
- 用户未提前登录苍穹环境
- 心跳页未授信
- 心跳页域名与登录环境不一致
- 心跳页使用浏览器与登录环境不一致

建议排查顺序：
1. 先确认目标环境已经登录
2. 再确认心跳页被成功打开
3. 检查域名是否一致
4. 检查是否使用同一浏览器

修复建议：
- 优先统一浏览器和域名
- 升级到较新的 VSCode 版本并校正外链打开方式

回归验证：
- 心跳页计数持续增长
- 页面触发事件后能进入断点

### 4. 单据无法选择已有插件

常见现象：
- 脚本已上传，但单据注册插件时下拉列表看不到

可能根因：
- 插件基类与业务场景不匹配

建议排查顺序：
1. 确认当前注册的是单据插件、表单插件还是其他插件
2. 检查代码中继承的基类
3. 检查模板是否误用默认的 `AbstractFormPlugin`

修复建议：
- 把插件基类改成对应场景的基类，例如单据场景使用 `AbstractBillPlugIn`

回归验证：
- 已上传插件可在目标场景中被正常选择
- 绑定后对应事件可正常触发

### 5. 长整型精度丢失

常见现象：
- 主键或大整数查询结果异常
- 明明是同一个 ID，查询或比较结果却不一致

可能根因：
- 把超出安全范围的整数按 `Number` 处理

建议排查顺序：
1. 检查字段是否可能超出 `2^53-1`
2. 检查是否错误使用了 `number`
3. 检查 VSCode 或项目配置是否支持 `BigInt`

修复建议：
- 改用 `BigInt`
- 如编辑器不支持，调整 `tsconfig.json` 目标版本

回归验证：
- 大整数在显示、比较、查询、回写时都不丢精度
- `QFilter` 查询和主键读取结果一致

### 6. 序列化、日期、default 方法与 QFilter 类型问题

这四类问题经常一起出现，因为它们都属于“声明看起来像 TypeScript，运行时其实受 Java 对象语义约束”的问题。

共同排查原则：
- 先区分当前值到底是 JS 原生对象还是 Java 对象
- 再决定使用 `JSON` 还是 `SerializationUtils`
- 日期比较优先用 `compareTo` 或 `getTime`
- 接口 default 方法调用前先判空，必要时在实现类中补实现
- `QFilter` 单个条件内部要用 Java 集合类型，多条件组合再用 TS 数组

对应详细文档：

- [4.4.7日期问题](./common-issues/4.4%E5%B8%B8%E8%A7%81%E9%97%AE%E9%A2%98/4.4.7%E6%97%A5%E6%9C%9F%E9%97%AE%E9%A2%98.md)
- [4.4.8接口default方法](./common-issues/4.4%E5%B8%B8%E8%A7%81%E9%97%AE%E9%A2%98/4.4.8%E6%8E%A5%E5%8F%A3default%E6%96%B9%E6%B3%95.md)

## 推荐提问格式

当用户带着报错来问时，优先补齐以下信息：

- 当前脚本属于哪种插件类型
- 触发事件或调用入口是什么
- 完整报错信息或截图
- 相关代码片段
- 使用到的 SDK、开放类、Java 类型或过滤条件
- 问题发生在编辑期、保存时、运行时、调试时还是部署后

## 使用建议

- 先看本文件，定位问题属于哪一类
- 再打开 `core/docs/common-issues/4.4常见问题/` 下的单篇文档看细节
- 如果单篇文档仍不能解释现象，回到“总体排查顺序”重新检查上下文、环境、绑定关系和类型边界
