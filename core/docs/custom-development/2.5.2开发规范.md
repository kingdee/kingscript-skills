# KingScript开发规范
## 1. 概述

### 1.1 文档目的
本文档旨在为二次开发人员提供一套清晰、可行的 KingScript 脚本编码、开发与 API 调用规范。通过遵循本规范，可以提高代码的可读性、可维护性和健壮性，同时确保团队协作的顺畅，并最大化脚本在苍穹平台上的运行效率和安全性。

### 1.2 核心原则
* **一致性（Consistency）**：所有开发者应遵循相同的代码风格和命名约定，以减少认知负担。
* **清晰性（Clarity）**：代码应自解释，避免模糊不清的命名和逻辑，必要的复杂逻辑需辅以详尽的注释。
* **健壮性（Robustness）**：脚本应具备完善的异常处理机制，能优雅地处理各种预期和非预期的错误。
* **高效性（Efficiency）**：充分利用苍穹平台提供的特性，编写高性能的脚本，避免不必要的开销。
* **安全性（Security）**：在与外部系统交互和处理敏感数据时，必须遵循安全规范，防止数据泄露和注入攻击。

### 1.3 适用范围
本规范适用于所有在苍穹平台后端执行的 KingScript 脚本开发的用户。

### 1.4 名词解释
* 轻脚本，即KingScript，是苍穹平台采用TypeScript语法运行在JVM中的服务端脚本语言。
* 模块，模块（Module），独立文件封装代码逻辑，一个模块可以定义多个export，通过export暴露接口供其他模块import使用。
* 插件，苍穹平台单据、报表、列表、OpenAPI、调度等二次开发的代码方式，通过插件可以实现个性化功能。
* 扩展点，苍穹平台针对业务功能暴露的扩展接口，二开可以通过扩展点进行业务扩展开发。

## 2. 开发与构建流程规范

### 2.1 开发行为规范
* 【强制】二次开发人员应使用苍穹平台提供的WEB IDE 或 VSCode插件 来编写、调试和管理 KingScript 脚本。这两种方式均提供了语法高亮、自动完成和实时错误提示等功能，是轻脚本开发的主要工具。

* 【强制】禁止禁用或删除标准产品的插件及扩展点脚本

说明：标准产品功能不满足业务需求时，可以创建自定义脚本进行二开。

* 【强制】禁止使用标准产品未开放的SDK方法

说明：脚本SDK是标准产品为脚本开放的标准能力，使用未开放的接口或功能可能导致脚本异常。

### 2.2 提交规范
* 【强制】开发人员将 KingScript 脚本源码，通过 VSCODE插件 提交到KingCode 代码仓的ks文件夹中

### 2.3 构建规范
* 【强制】二开人员使用 DCS（协同开发平台）进行构建

### 2.4 代码扫描与处理规范
* 【强制】二开人员使用 DCS（协同开发平台）启动构建后，DCS平台自动对 KingScript 脚本进行扫描，扫描规则请参考第 3\~5 部分相关的编码规范。
* 【强制】开发者必须根据代码扫描报告，修复其中发现的问题，如果问题未修复，质量门禁会禁止补丁推送到客户环境。

### 2.5 部署
* 【强制】开发者使用DCS（协同开发平台）将KingScript补丁包推送到天梯进行部署

***Tips：上述 DCS 启动构建、查看扫描报告等，请登录 DCS，在产品首页的用户指引中查看***

### 2.6 生产运行
* 【强制】禁止在生产环境修改脚本

说明：为保证脚本一致性和系统稳定性，生产环境中禁止修改脚本，尽可能在开发环境修改，通过部署方式推送至测试环境、用户验证环境、生产环境。

* 【强制】禁止在生产环境调试脚本

说明：生产环境调试脚本，可能影响服务稳定导致业务异常，还可能导致敏感数据泄露引发安全事故。

## 3. 编码及命名规范
说明：本规范为架构规范的开发视图部分，开发人员通过此规范，可以了解开发环境的项目结构和开发过程中的各种命名。

### 3.1 工程命名目录结构规范
* 【强制】统一为ks，已与 VSCODE插件 强集成

### 3.2 工程目录结构规范
* 【强制】遵循苍穹平台的标准目录结构，保持文件和模块的逻辑组织。不同的功能模块应存放在独立的目录下，以便于管理和维护。
```
ks
├─.setting              <-- IDE自动生成
├─.vscode               <-- IDE自动生成
├─node_modules          <-- 脚本依赖包目录
│  └─@cosmic            <-- 苍穹平台API依赖包目录
│      ├─bos-core
│      ├─bos-framework
│      ├─bos-script
│      ├─bos-test
│      └─bos-util
└─src                   <-- 二开脚本目录
```

### 3.3 脚本文件命名规范
* 【强制】所有二开脚本放在src目录下，且放入@{产品名}/{云}/{应用}/{脚本文件名}.ts

* 【强制】**产品名**按产品取值
  * 苍穹使用@cosmic
  * 星空旗舰版使用@galaxy
  * 星瀚使用@constellation

* 【强制】**脚本文件名** 应遵循大驼峰式（PascalCase），可与类名一致
* **示例**：`@cosmic/democloud/demoapp/PaymentBill.ts`
```
ks
├─.setting
├─.vscode
├─node_modules
│  └─@cosmic
│      ├─bos-core
│      ├─bos-framework
│      ├─bos-script
│      ├─bos-test
│      └─bos-util
└─src                               <-- 二开脚本目录
    └─@cosmic                       <-- @{产品名}
        └─democloud                 <-- {云}
            └─demoapp               <-- {应用}
                └─PaymentBill.ts    <-- {脚本文件}
```

### 3.4 模块导入导出规范
#### 3.4.1 模块导入
* 【强制】如果import的多个模块在一个module中时，尽量一个import语句完成，避免出现多个import语句。
```typescript
【正例】
import { EventObject, HashMap } from "@cosmic/bos-script/java/util";
```
```typescript
【反例】
import { EventObject } from "@cosmic/bos-script/java/util";
import { HashMap } from "@cosmic/bos-script/java/util";
```

* 【强制】插件导入名称为plugin
```typescript
// 插件导入名称为plugin
import { plugin } from "@cosmic/mycloud/myapp/myplugin";
```

* 【强制】扩展点导入名称为ext
```typescript
// 扩展点导入名称为ext
import { ext } from "@cosmic/mycloud/myapp/myext";
```

* 【强制】工具类导入名称为util
```typescript
// 工具类导入名称为util
import { util } from "@cosmic/mycloud/myapp/myutil";
```

#### 3.4.2 模块导出
* 【强制】插件导出名称为plugin
```typescript
import { AbstractFormPlugin } from "@cosmic/bos-core/kd/bos/form/plugin";
import { EventObject } from "@cosmic/bos-script/java/util";
class MyPlugin extends AbstractFormPlugin {
  registerListener(e: EventObject): void {
    super.registerListener(e);
  }
}
let plugin = new MyPlugin();
//【强制】导出插件实例，命名必须为plugin
export { plugin }
```

* 【强制】扩展点导出名称为ext
```typescript
import { ExtInterface } from 'ExtInterface_SDK_Path';
let ext : ExtInterface = {
  method(name : String) : String {
    return 'hello' + name;
  }
}
//【强制】导出扩展点，命名必须为ext
export { ext };
```

* 【强制】工具类导出名称为util
```typescript
let util = {
  method(name : String) : String {
    return 'hello ' + name;
  }
}
//自定义脚本，如工具类等，导出名称不限制
export { util };
```

### 3.5 类与接口命名规范
* 【强制】类和接口名应使用大驼峰式（PascalCase），并采用名词或名词短语。
* **示例**：`SalOrderService`, `CustomerValidator`
```
interface IAnimal {
  name: string
}

class Dog implements IAnimal{
  constructor(){
    this.name = "dog";
  }
}
```

### 3.6 变量、函数与方法命名规范
* 【推荐】变量、函数和方法名应使用小驼峰式（camelCase）。
* 【推荐】常量，应使用全大写字母，并用下划线（`_`）分隔。
* **示例**：`const MAX_ORDER_AMOUNT = 10000;`, `let orderPrice = 200;`, `saveData()`

## 4. 代码风格及类型系统规范
### 4.1 代码风格规范

#### 4.1.1 缩进与空格
* 【推荐】使用 TAB 作为缩进。在运算符（如 `+`, `-`, `=`, `===`）前后都应加空格。

#### 4.1.2 换行与行长
* 【推荐】每行代码长度应不超过 120 个字符。过长的语句应在适当的位置进行换行，以提高可读性。

#### 4.1.1 注释风格
* 【推荐】
  * **文件顶部**：简要说明文件用途。
  * **函数/方法**：使用 JSDoc 风格的块注释（`/** ... */`），描述其功能、参数、返回值和可能抛出的异常，这有助于生成文档。
  * **行内注释**：使用双斜杠（`//`），解释代码中不清晰的部分或业务逻辑。

#### 4.1.4 语句结束符
* 【强制】每条语句都必须以分号（`;`）结束

### 4.2 类型系统与变量声明

#### 4.2.1 变量声明
* 【推荐】常量使用 `const`，变量使用 `let`

#### 4.2.2 类型声明
* 【推荐】函数参数和返回值应显式声明类型。为复杂类型创建接口（`interface`）或类型别名（`type`）。尽量避免使用 `any`，除非在特定且有注释的情况下。

### 4.3 函数与类

#### 4.3.1 函数
* 【推荐】内部匿名函数推荐使用箭头函数（`=>`）
* **错误示例**：
```
class MyClass {
  message = 'Hello';
  logMessage() {
    doTask(function() {
      console.log(this.message); // this 是 undefined
    });
  }
}
```
* **正确示例**：
```
class MyClass {
  message = 'Hello';
  logMessage() {
    doTask(() => {
      console.log(this.message); // this 正确地指向 MyClass 实例
    });
  }
}
```

#### 4.3.2 类
* 【推荐】使用访问修饰符（`public`, `private`, `protected`）。对于在构造函数中初始化后不应再被修改的属性，使用 `readonly` 关键字来确保不可变性。

### 4.4 异常处理规范

#### 4.4.1 自定义异常
* 【强制】不支持自定义异常类型，所有异常都应使用平台提供的标准异常类型。

#### 4.4.2 错误日志记录
* 【推荐】在捕获到异常时，应通过 `logger.error("错误分析描述", e)` 记录详细的错误日志，包括异常信息、堆栈跟踪和相关业务上下文，以便于问题排查。

## 5. 苍穹平台API脚本调用规范

### 5.1 安全规范

#### 5.1.1 多租户架构数据隔离规范

* 【强制】禁止使用系统环境变量、数据中心上下文、MC配置上下文、数据库配置器、数据源工厂等工具类，访问多租户信息，避免引发跨租户访问数据
* 【强制】禁止通过反射、字节码等操作，破坏苍穹架构约束，避免引发跨租户访问数据
* 【强制】禁止直接使用第三方工具类访问中间件，如Redis、zookeeper、MQ等，避免引发跨租户访问数据
* 【强制】禁止直接创建JDBC数据库连接，避免引发跨租户访问数据

#### 5.1.2 敏感信息处理

* 【强制】代码中不能存在硬编码敏感信息，如账号、密码、http外链、ftp外链等；配置文件、SQL脚本等文件中不能存在明文密码等敏感信息
* 正例：
  * 将密码等敏感信息加密后存入MC，代码再获取MC的值，通过解密得到具体的密码值，加密算法建议采取AES256或者SHA256。
  * 所有的外链都必须写在配置文件，或者存放数据库中，产品默认的外链需要说明原因

* 【强制】页面不能明文显示敏感信息
* **说明**：具体请[使用平台统一能力隐私中心进行设置](https://vip.kingdee.com/knowledge/319849321624046848?productLineId=29&isKnowledge=4&lang=zh-CN#1)

#### 5.1.3 第三方系统交互
以下规范基于苍穹的开发可以不关注，苍穹已经做了对应的防范并已通过安全测试，但与第三方系统交互，第三方API或开放API时，需要特别关注。
* 【推荐】与第三方系统进行交互时，对传递的参数需要做校验，如果不合法，可直接拒绝请求
  * 如果是数字类型，判断是否为数字，如果不是，直接拒绝请求；
  * 如果是有格式的类型（比如日期、email、电话、身份证号码等等），判断是否符合格式，如果不符合格式，直接拒绝请求；
  * 如果是没有特殊格式的字符串类型，需要对字符串长度做校验。
  * 如果长度大于数据库该字段的定义长度，直接拒绝请求（比如姓名，数据库定义的是 varchar(12)，如果输入超过12个字符，直接拒绝请求）。

* 【强制】禁止使用Statement拼接导致的SQL注入攻击
* 说明：使用PrepareStament，PrepareStament可以有效防御SQL注入攻击，其原理是：MySQL执行树会根据预设的参数做转义，把注入的SQL当作一个参数执行，而不会当作语句执行

#### 5.1.4 XXE外部实体注入
* 【强制】禁用xml外部实体；
* 【强制】过滤xml外部实体的关键字，过滤<!DOCTYPE>, <!ENTITY>, SYSTEM等。

#### 5.1.5 跨站脚本（XSS）
* 【推荐】对数据将要置于的HTML上下文（包括主体、属性、JavaScript、CSS或URL）的所有不可信数据进行恰当的转义（escape），例如：对特殊符号< >进行html编码输出。
* 说明：基于苍穹控件的开发可以不关注以下防范措施，苍穹已经做了对应的防范并已通过安全测试，但使用自定义控件时，需要特别关注

#### 5.1.6 附件功能开发规范
* 【强制】**禁止单据绑定临时附件**
  * **说明**：
    * 单据未保存时，上传的附件为临时附件，后续系统会自动清理临时附件；单据的保存功能会**调用临时附件持久化接口，将临时附件转为正式附件**；
    * 定制开发有些功能不调用单据保存，自行处理单据与附件的关系时，直接将临时附件记录在关系表中，系统清理临时附件后单据附件丢失；
    * 定制开发自行处理单据与附件的关系时，应先调用附件的持久化接口将临时附件转为正式附件，具体请参考[临时附件持久化接口](https://dev.kingdee.com/sdk/Cosmic%20V7.0.1/index.html?nav=class&module=kd.bos.svc.attachment&package=kd.bos.servicehelper&name=AttachmentServiceHelper#saveTempToFileService-java.lang.String-java.lang.String-java.lang.String-java.lang.Object-java.lang.String-boolean-boolean-)
* 【强制】**禁止定制开发自己写SQL删除附件与单据的关联关系**
  * **说明**：定制开发代码要删除附件时，请调用附件删除接口，该接口会在删除单据与附件的关联时，根据附件是否还被其他单据引用，来确定附件的删除，不需要定制开发代码做特殊处理。具体请参考[附件删除接口](https://dev.kingdee.com/sdk/Cosmic%20V7.0.1/index.html?nav=class&module=kd.bos.svc.attachment&package=kd.bos.servicehelper&name=AttachmentServiceHelper#remove-java.lang.String-java.lang.Object-java.lang.Object-)
* 【强制】调用文件服务接口直接上传附件，**PATH路径建议体现附件所在单据**（租户Id\_数据中心\_年月日\_应用Id\_表单标识\_单据主键\_附件名），便于后续对问题附件做追踪。
* 【强制】定制开发使用自定义组件操作附件时，**调用接口需传入表单标识参数（fid）**，附件操作权限管控才能生效
  * **说明**：具体请参考[附件操作支持权限控制](https://vip.kingdee.com/knowledge/381378954285846016?productLineId=29&isKnowledge=2&lang=zh-CN)

### 5.2 性能规范

#### 5.2.1 事务规范
* 【强制】**避免大事务**
  * **说明**：
    * 大事务执行效率低，可能造成其他事务长时间锁等待和锁冲突，进而发生阻塞或者死锁
    * 优化方式： 需要对事务中耗时的操作进行拆分处理，将大事务拆成多个小事务

#### 5.2.2 数据访问规范
* 【推荐】BusinessDataServiceHelper和QueryServiceHelper使用规范**

| 方法 | 适用场景 | 缓存 | 数据结构 | 能否保存 | 备注 |
| :---- | :---- | :---- | :---- | :---- | :---- |
| BusinessDataServiceHelper.loadSingle | 取单张单据 | 无 | 单据结构一致 | 能 | 实时读取数据库，传入pk不存在时，会中断输出单条符合单据结构的数据包 |
| BusinessDataServiceHelper.loadSingleFromCache | 取单条基础资料 | 有 | 单据结构一致 | 否 | 优先从缓存中读取，缓存没有则实时取数据库，并压入缓存 读取的数据包，经过缓存处理，少了状态信息 |
| BusinessDataServiceHelper.load | 取一批单据   | 无 | 单据结构一致 | 能 | 先根据条件取pks，再根据pks实时取数，没有符合条件数据时返回空集合，不报错 |
| BusinessDataServiceHelper.loadFromCache | 取一批基础资料 | 有 | 单据结构一致 | 否 |  |
| QueryServiceHelper.query | 取一批数据   | 无 | plainObject | 否 |  |
| QueryServiceHelper.queryOne | 取一行数据 | 无 | plainObject | 否 |  |
| QueryServiceHelper.queryDataSet  | 取个别字段值 | 无 | plainObject | 否 | 流式取数，性能最快 |

* 【强制】所有脚本必须使用KSQL语法，禁止使用方言
* 【强制】**大数据量查询必须添加过滤条件，禁止无过滤条件查询**
* 【强制】**自建二开表单或扩展标准单据新增二开字段，涉及到高频查询一定要考虑索引，避免全表扫描**
* 【强制】**批量操作**：避免在循环中逐条操作数据库，优先使用批量数据访问 API，以减少数据库连接开销。
* 【强制】**大数据量的查询、写入及计算等操作必须进行分批处理**
* 【推荐】**按需查询**：查询数据时，只选择需要的字段，避免使用 `*`，减少数据传输量。
* 【推荐】**基于性能因素考虑，数据查询方法选择：**
  * 对于存在大数据表的查询，应该使用QueryServiceHelper.queryDataSet；
  * 不查询大量数据才可以用QueryServiceHelper.query；
  * 如果能用QueryServiceHelper.query满足功能，就不要用BusinessDataServiceHelper.load，这个方法对性能影响更大，通常在需要通过ORM对象保存回数据库时才用。
* 【强制】数据查询取数及过滤条件，访问字段应避免出现4层及以上
* 说明：访问字段层数过多易产生笛卡尔积爆炸，从而导致性能骤降，可通过合理创建索引、分步查询、临时表、冗余字段等技术手段来规避问题

#### 5.2.3 循环操作
* 【强制】禁止在循环中访问数据库
* 说明
  * 如果有批量接口，建议采用批量接口访问
  * 代码优化思路：可以先提取查询条件，一次性批量查询，再在循环中处理对应的业务逻辑

* 【强制】禁止在循环中调用view.updateView()
* 说明：可在循环结束后针对修改的数据进行局部刷新

* 【强制】禁止对大数据包(>100)循环使用model.setValue更改数据
* 说明：如需修改，建议使用property.setValueFast

* 【强制】禁止循环中调用DB.***()
* 【强制】禁止循环中调用ORM.create
* 【强制】禁止循环中调用DispatchServiceHelper.invoke***()

#### 5.2.4 缓存使用规范
* 【推荐】**使用分布式缓存，禁止缓存大对象**
* **说明**：由于缓存使用redis作为框架的缓存底层 ，如果存储的内容过多，会导致redis产生bigkey，阻塞redis主线程，从而导致redis存在访问性能问题
* 【推荐】**按场景合理使用缓存访问数据**
* **说明**：
  * 频繁访问的数据应增加缓存，缓存的对象类型应考虑最大程度共享；
  * 针对总数据量小于1000条并且改动不频繁的基础资料的查询，可以合理使用BusinessDataServiceHelper.loadFromCache提升效率，对于数据量超过1000的，谨慎使用BusinessDataServiceHelper.loadFromCache，如果确定要使用需要找相关领域架构师进行评审

#### 5.2.5 资源管理
* 【强制】为了保证资源的确定性释放，所有实现了资源关闭接口（例如：DataSet、DataSource 等需要调用 close() 方法）的对象，在 KingScript 中必须使用标准的 try...finally 结构进行包裹，并在 finally 块中显式调用资源的 close() 方法。

* **错误示例**：
```
// 错误：资源管理不当，未确保 close() 被调用，可能导致连接泄露或资源耗尽
const dataSet = new DataSet(query);
// ... 操作 dataSet
// ... 可能在执行过程中抛出异常，导致 dataSet.close() 永远不会执行
dataSet.close();
```

* **正确示例**：
```
// 正确：使用 try...finally 结构，确保资源在任何情况下都能关闭
let dataSet: DataSet | undefined; // 声明时需考虑类型
try {
  dataSet = new DataSet(query);
  // 操作 dataSet
} catch (e) {
  // 异常处理
  console.error('查询数据失败', e);
} finally {
  if (dataSet) {
    // 确保 close() 在 try 块结束后一定被调用
    dataSet.close();
  }
}
```

#### 5.2.6 日志规范
* 【强制】**除一些特殊工具外，所有程序日志必须使用kd.bos.logging.Log接口**

* 【强制】**日志打印大列表对象或完整动态对象时，需要评估后按需打印**
* **说明**：
  * 打印列表对象时需要评估，如果列表比较大或者包含了完整的动态对象不要直接打印该列表的内容，要按需打印；
  * 打印动态对象时不能打印完整的对象内容，要按需打印。

* 【强制】**禁止打印日志时，使用json序列化SerializationUtils.toJsonString(args)打印页面对象或数据对象的json串**
* **说明**：json序列化方法非常消耗cpu

* 【推荐】**输出日志先判断当前日志级别是否开启**
* **说明**：该操作可以减少系统性能开销
* 【推荐】**日志输出，有异常时应带上异常对象**
* **说明**：如果没有异常对象信息，会导致对日志进行错误分析时无法定位错误原因。
  * **正例**：使用logger.error("错误分析描述", e) 形式，加上错误分析描述方便从错误日志中查询特定问题的日志。
  * **反例**：直接调用Throwable.printStackTrace()。
* 【推荐】**正确设置日志级别：Debug、Info、Warn、Error**
* **说明**：如不是错误或者异常等，慎用error级别的日志。
* 【推荐】**谨慎记录日志**
* **说明**：大量地输出无效日志，不利于系统性能，也不利于快速定位错误。
* 【推荐】**对于一些接口调用或者操作的关键信息，开发过程中应在适当的位置将日志输出，便于后续排查问题**

#### 5.2.7 多线程规范
* 【强制】**KingScript脚本不允许使用多线程，如果业务逻辑需要异步执行，可以使用调度中心而不是多线程**。调度中心做了分布式管理，它能确保在集群环境下任务能在正确的节点正确地被调度执行
* **说明**：[调度中心使用指南](https://vip.kingdee.com/knowledge/specialDetail/389113676499986688?productLineId=29&lang=zh-CN) 供参考
