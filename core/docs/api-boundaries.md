# Kingscript API Boundaries

本文件用于约束 skill 的边界，避免模型乱编。

## 必须明确的规则

- 不允许编造 Kingscript API、事件名、上下文对象结构。
- 不允许默认某个后端 Java 能力已经通过 SDK 注解开放。
- 不允许忽略权限、租户、账套、组织、上下文边界。
- 不允许把 TypeScript 声明文件当作运行时必然可用的事实。
- 不确定时必须明确标记为“假设方案”。

## 事实优先级

当多个来源不一致时，优先级按下面顺序判断：

1. 真实运行时行为、日志、可复现结果
2. 平台明确开放的 SDK 文档和在线 Javadoc
3. 本地 TypeScript 声明和生成清单
4. 已验证的示例和案例
5. 基于经验的推断

越靠后的来源，越不能单独作为“确定可用”的依据。

## 常见边界类型

### 1. 开放能力边界

- 只有被平台显式开放的 Java 类、方法、接口和扩展点，才应视为可供二开使用
- 即使某个类在内部 Java 代码中存在，也不代表 Kingscript 可以直接使用
- 即使 `.d.ts` 中出现了某个声明，也不能自动推出它在当前运行环境里一定可用

### 2. 插件类型边界

- 插件基类必须和业务场景匹配
- 表单、单据、列表、报表、操作服务、调度等插件类型不能混用
- 事件是否存在、是否触发，取决于当前插件类型和绑定场景

例如：

- 单据场景常见基类是 `AbstractBillPlugIn`
- 普通表单场景常见基类是 `AbstractFormPlugin`
- 操作服务校验等场景通常走 `AbstractOperationServicePlugIn` 及其相关校验扩展

### 3. 生命周期边界

- 同一个对象在不同事件阶段的可见性不同
- 过早阶段可能还没有数据模型或界面控件
- 过晚阶段再修改数据，可能已经错过联动或校验时机

判断 API 是否能用时，不能只看“有没有这个方法”，还要看“当前时机能不能调用”。

### 4. 权限与隔离边界

- 当前用户是否有权限
- 当前租户、账套、组织是否正确
- 当前登录态和上下文是否完整

很多“查不到数据”“服务调用失败”“同样脚本在不同环境行为不一致”的问题，本质上不是 SDK 不存在，而是上下文边界没有满足。

### 5. 运行时类型边界

Kingscript 运行时经常需要处理 Java 对象与 JS 对象的边界差异，尤其要注意：

- `BigInt`、`BigDecimal`、长整型主键
- `Date`
- `ArrayList` 等 Java 集合
- `QFilter` 的参数类型
- `SerializationUtils` 与原生 `JSON`
- Java 接口 default 方法

常见原则是：

- 编辑器看起来像 TypeScript，不代表运行时对象就是原生 TS/JS 对象
- 单个 API 的参数要优先按运行时实际要求来判断
- 类型不确定时，优先查 FAQ、知识卡、Javadoc 和最小示例

## 声明与运行时不一致时的判断顺序

当出现“声明存在，但运行时报错”时，优先按下面顺序判断：

1. 当前能力是否真的属于已开放 SDK
2. 当前插件类型和生命周期是否匹配
3. 当前上下文对象是否存在且已初始化
4. 当前参数类型是否符合 Java 运行时要求
5. 当前用户、租户、账套、组织、权限是否满足
6. 当前环境版本是否与声明来源一致

如果以上仍不能确认，就必须把答案降级为“有界方案”，而不是继续假装确定。

## AI 输出时必须遵守的约束

- 先说清楚当前回答基于哪一层来源
- 如果依赖假设，必须明确写出假设条件
- 如果涉及未确认的开放能力，必须直接说“尚未确认是否已开放”
- 如果问题本质上是运行时边界问题，不能只给代码，还要提示检查项

## 可以安全给出的内容

- 基于已有文档、示例、知识卡整理出的用法说明
- 明确标注前提条件的代码草稿
- 基于 FAQ 和排障手册的根因分析
- 基于 `.d.ts` 与 Javadoc 的结构化解释

## 不可以安全给出的内容

- 凭空发明的事件名、上下文对象字段、插件方法
- 默认可访问的私有后端类或内部服务
- 未核实就宣称“运行时一定支持”的结论
- 绕过权限、租户、账套、组织限制的建议

## 推荐使用方式

- 先读 `content-map.md` 确认内容位置
- 先读 `concepts.md` 建立基本心智模型
- 涉及 SDK 时先走索引，再查知识卡，再降级到清单、本地声明和 Javadoc
- 涉及报错时优先结合 `troubleshooting.md` 和 `common-issues/`
